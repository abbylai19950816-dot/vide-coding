# 2026-05-16 資料健康檢查加入學員查詢索引修復入口

## 需求摘要

延續「多老師版本前的資料一致性基礎」，把學生端查不到方案的原因納入管理員資料健康檢查。管理員不應只看到一顆「強制重建學員查詢」按鈕，而要知道何時該用、為什麼要用。

## 變更內容

- `active/gyrobooking_current/github_pages/admin.html`
  - 資料健康檢查改為 async，可在管理員端檢查 `student_lookup/{hash}` 是否存在。
  - 新增檢查：
    - 學員有有效票券，但 `student_lookup` 不存在。
    - `student_lookup` 剩餘堂數與管理員端推算不同步。
    - 有剩餘票券，但票券課程類型無法對應目前公開課表課程類型。
  - lookup 缺失或不同步的項目下方會顯示「重建學員查詢」修復入口。
  - 從檢查結果觸發重建後，系統會自動重跑資料健康檢查。
- `admin.html`
  - 已同步 GitHub Pages 根目錄入口檔。
- `docs/ssot/data_model.md`
  - 補上 lookup 檢查範圍與維運修復入口規則。
- `docs/ssot/cost_model.md`
  - 補上 lookup 檢查的管理員端維運讀取成本說明。

## Firestore 讀寫影響

- 學生端：沒有新增讀取或寫入。
- 管理員端資料健康檢查：會按預期 lookup 文件逐筆讀取 `student_lookup/{hash}`，用來判斷公開索引是否缺失或不同步。
- 管理員端修復入口：沿用既有 `lowCostSyncPublicMirror(..., { force: true })`，低頻手動重寫公開索引。

## 驗證

- 已同步 active 與根目錄 `admin.html`。
- 待部署後可在設定頁執行資料健康檢查，確認 lookup 問題下方是否出現「重建學員查詢」。

## 後續風險

- 課程類型無法對應的檢查依目前公開課表與票券資料推算；若老師只是尚未開課，這是提示而非資料錯誤。
- 未來可再加入「票券類型對應設定」工具，讓管理員明確修正舊票券與新課程類型的對應關係。
