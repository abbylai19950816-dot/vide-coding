# 使用者體驗改善路線圖

## 文件目的

本文件整理 Gyrobooking 預約系統日後可優先改善的 UX 方向。現階段成本策略以 Firebase 免費版為主，目標是在不明顯增加 Firestore read/write 成本、不引入必要付費後端的前提下，讓學生端更安心、管理端更有效率、維護者更容易判斷每次更新的價值。

## UX 原則

- 學生端要像「3 分鐘內完成」的輕流程：看懂方案、送出購課、查詢剩餘堂數、完成預約。
- 管理端要像「每天重複使用的工作台」：快速掃描、快速處理、少打字、少誤操作。
- 所有提示要回答使用者當下的疑問：現在發生什麼、下一步做什麼、如果失敗要找誰。
- 優先改善前端狀態與文案，不優先增加 Firestore 讀取。
- 任何 UX 改動都要同時檢查 `docs/firestore_cost_guardrails.md`。
- 現階段優先使用 Firebase Spark/free tier 可承受的方案；除非使用者明確同意升級成本，不把 Cloud Functions、外部付費服務或高頻即時監聽當成預設方案。

## 學生端短期改善

### 1. 首頁流程改成更明確的「三步驟」

目前學生端有 `購買方案` 與 `預約時間` 兩個 tab。建議在購買頁頂部用簡短流程說明：

```text
1. 選擇課程與方案
2. 填寫聯絡資料
3. 加 LINE / 等待老師確認後預約
```

效益：

- 降低第一次使用者不知道「送出後會發生什麼」的不安。
- 減少重複送出或私訊詢問。

成本影響：

- 純前端文案，無 Firestore 影響。

### 2. 購課送出後顯示更完整的下一步

成功頁建議顯示：

- 已收到申請。
- 老師會如何確認付款。
- 若想更快處理，請點 LINE 按鈕。
- 可以稍後到「預約時間」用姓名與手機查詢。

效益：

- 讓使用者知道購課不是即時開通時，不會誤以為失敗。

成本影響：

- 純前端文案，無 Firestore 影響。

### 3. 查詢剩餘堂數時增加「查不到」的原因提示

目前查不到可能是：

- 姓名或手機格式不一致。
- 老師尚未確認付款。
- 票券已過期或堂數用完。
- 系統尚未同步。

建議查不到時顯示可操作訊息：

```text
查不到可預約方案。請確認姓名與手機是否與購買時一致。
如果剛完成付款，可能需要老師確認後才會開通。
```

並提供 LINE 聯絡按鈕。

成本影響：

- 不增加讀取。仍只 get `student_lookup/{hash}`。

### 4. 預約成功後顯示預約摘要

成功預約後顯示：

- 課程名稱
- 日期
- 時間
- 剩餘堂數
- 提醒文字，例如「若需取消或改期，請聯絡老師」

效益：

- 使用者有明確完成感，也能截圖保存。

成本影響：

- 使用 transaction 回傳資料即可，不需額外 read。

### 5. 時段列表增加可掃描性

建議時段卡片固定呈現：

- 日期與星期
- 時間
- 課程類型
- 剩餘名額
- 是否即將額滿

若同一天沒有時段，空狀態要說「這天沒有可預約時段」，而不是只留空白。

成本影響：

- 使用既有 `public_booking/state.slots`，不增加 read。

## 學生端中期改善

### 1. 預約前確認底部 sheet

在點擊預約後，使用底部確認 sheet 顯示本次預約內容與扣堂數：

```text
確認預約
課程：墊上嬋柔
時間：5/20（三）19:00
本次扣除：1 堂
預約後剩餘：3 堂
```

效益：

- 避免誤點。
- 讓扣堂規則更透明。

成本影響：

- 只用本機已載入資料與 transaction 回傳結果。

### 2. 加入「我的預約」摘要

學生查詢成功後，可以顯示近期已預約課程。

注意：

- 短期不要為此新增額外 private read。
- 可以從 `student_lookup` 增加 minimal upcomingBookings summary，但要控制欄位大小。

建議欄位：

```js
upcomingBookings: [
  { slotId, date, time, typeId, typeName }
]
```

成本影響：

- 仍讀同一份 `student_lookup/{hash}`。
- 會增加 lookup 文件大小，需限制只放未來 3 至 5 筆。

### 3. 取消/改期入口先做「聯絡老師」

自助取消會牽涉退款堂數、名額釋放、多文件 transaction。若尚未有 Cloud Functions 或完整 rules，不建議先做全自動取消。

建議短中期做：

- 顯示已預約課程。
- 提供「我要改期 / 取消」LINE 按鈕。
- 自動帶入文字模板。

成本影響：

- 無額外 Firestore 寫入。

## 管理端短期改善

### 1. 首頁 Dashboard

管理端登入後建議先顯示今日摘要：

- 今日課程數
- 今日預約人數
- 待確認收費
- 即將到期票券
- 未同步或異常票券

效益：

- 管理者不用每次切很多 tab 找待辦。

成本影響：

- 使用管理端初始化已讀入的 `/data/*` memory cache 計算，不新增 Firestore read。

### 2. 待辦中心

建立「待處理」區塊，集中顯示：

- 未收費 / 待確認付款
- 票券已付款但未開通
- 有預約但票券扣堂異常
- `unresolvedTickets`

效益：

- 降低漏處理與資料不同步。

成本影響：

- 使用既有 admin cache。

### 3. 危險操作改成維護工具區

目前 console 工具有：

- `resetAllTestData()`
- `reconcilePaidPaymentsToTickets()`
- `repairBookingsToStudentRecords()`
- `deleteAllPurchaseRequests()`

建議搬到管理端「維護工具」頁：

- 每個工具先 dry-run。
- 顯示預估影響筆數。
- 需要二次確認。
- 執行後寫入維護紀錄。

效益：

- 減少只有開發者知道 console 指令的狀況。
- 降低誤刪資料風險。

成本影響：

- dry-run 使用本機 cache。
- 實際執行才寫 Firestore。

### 4. 管理端表格搜尋與篩選

建議所有列表優先支援：

- 搜尋姓名 / 手機 / LINE。
- 篩選課程類型。
- 篩選票券狀態：有效、即將到期、已用完、過期。
- 篩選付款狀態：未收、已收、待確認。

成本影響：

- 用本機 cache 篩選，不新增 Firestore read。

## 管理端中期改善

### 1. 學員詳情改成時間軸

學員詳情可以依時間呈現：

- 購買方案
- 收款
- 預約
- 扣堂
- 取消 / 補堂
- 出席紀錄

效益：

- 老師查問題時不用跨頁比對。

成本影響：

- 初期由現有 `students`、`tickets`、`payments`、`course_logs` 組合。
- 長期可改成 append-only `events` collection。

### 2. 課表批次操作

建議支援：

- 批次建立每週固定課程。
- 批次停課。
- 批次修改地點或老師備註。
- 遇到已有預約的時段時顯示影響名單。

成本影響：

- 管理端操作可接受較多 writes，但需顯示影響筆數。
- 寫入前要避免無差異重寫。

### 3. 票券異常檢查

管理端可提供「資料健康檢查」：

- 預約存在但學生 scheduledBookings 缺漏。
- 扣堂 logs 與 used/left 不一致。
- 付款已收但沒有票券。
- 票券 typeId 對不到目前課程類型。

成本影響：

- 使用本機 cache。
- 修復動作才寫入。

## 長期改善

### 1. 帳號化或魔法連結

目前用姓名 + 手機 hash 查詢，成本低但 UX 有限制。長期可考慮：

- 手機 OTP。
- Email magic link。
- LINE Login。

效益：

- 使用者不用反覆輸入。
- 可以更安全地顯示「我的預約」與取消功能。

成本影響：

- 不列為現階段優先項目。
- 會增加 Auth 與安全規則複雜度。
- 需要重新評估 Firestore rules 與成本。

### 2. 後端化 booking transaction

若正式營運規模變大，建議把預約、扣堂、取消、退款移到 Cloud Functions 或其他受控後端。

效益：

- Firestore rules 可大幅收緊。
- 學生端不需要公開寫 `/data/*`。
- 可做速率限制與完整審計。

成本影響：

- 不列為 Firebase 免費版優先方案。
- 會增加後端成本，但可換取安全性與一致性。

## 優先級建議

### 第一批：低成本、高感受

1. 購課流程三步驟說明。
2. 購課成功下一步文案。
3. 查不到方案的原因提示與 LINE 聯絡。
4. 預約成功摘要。
5. 管理端 Dashboard 草稿。

### 第二批：降低管理成本

1. 待辦中心。
2. 維護工具 UI 化。
3. 管理端搜尋與篩選。
4. 資料健康檢查。

### 第三批：產品成熟度

1. 我的預約摘要。
2. 改期 / 取消流程。
3. 學員時間軸。
4. 後端化 booking transaction。

## 驗收標準

每個 UX 改善都要回答：

- 使用者是否更清楚下一步？
- 是否減少老師人工確認或私訊溝通？
- 是否沒有增加學生端 Firestore reads？
- 是否有失敗狀態與可恢復路徑？
- 是否有同步更新 work log 與必要 SSOT？
