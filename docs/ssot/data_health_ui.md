# 資料健康檢查 UI SSOT

## 目的

資料健康檢查是管理員端的資料一致性報告工具，用來協助管理員理解「收費、票券、預約、行事曆、出缺勤、課程日誌、學員查詢」是否同步。

第一版原則是先產生報告與引導，不自動修改資料。需要修復時，必須由管理員明確按下對應按鈕。

## UI 分類

檢查結果分成三類：

- 嚴重：可能影響票券、預約或課程紀錄正確性的問題。管理員應先處理。
- 可修復：系統已有明確修復工具的項目，例如重建 `student_lookup` 或清除孤兒課程日誌成員。
- 人工確認：系統可以提示異常，但無法安全判斷是否應刪除或修改資料。管理員需先確認是否為測試資料或真實資料。

## 管理員操作順序

有問題時，報告應顯示「建議處理順序」：

1. 先處理嚴重項目。
2. 再處理可修復項目。
3. 最後重新執行資料一致性檢查。
4. 只剩人工確認時，先判斷資料來源，不確定時不要直接刪資料。

## 可修復按鈕

目前允許在報告卡片內直接顯示的修復動作：

- `previewStudentLookupRebuild('data-health')`：先預覽重建學員查詢索引，再由管理員確認執行。
- `previewCleanOrphanCourseLogMembers()`：先預覽課程日誌內不存在的學員成員，再由管理員確認清除。
- `renderTicketRecalcReport()`：查看票券重算報告。
- `previewReconcileHistoricalTicketLogs()`：先預覽歷史票券紀錄校準，再由管理員確認新增校準點。

這些按鈕只應出現在對應問題卡片內，避免管理員不知道下一步要做什麼。

## 正式維運修復規則

正式上線後，資料修復工具仍應保留，但定位是管理員或平台維運工具，不是日常操作功能。

所有會寫入或刪除資料的修復動作都必須符合：

- 先 dry-run / preview，列出會影響的資料量與範例。
- 說明「會改什麼」與「不會改什麼」。
- 管理員按下確認後才執行。
- 執行完成後要重新跑資料一致性檢查或更新報告。
- 高風險修復不可放在學生端，也不可在一般 render 流程自動執行。

## 成本與安全

這次 UI 優化不新增學生端 Firestore 讀寫，也不新增公開端資料讀取。

資料健康檢查仍屬於管理員端工具，讀取範圍可涵蓋管理員已載入或管理員端需要檢查的資料；不能移到學生端，也不能讓公開頁面執行批量資料檢查。

未來收緊 Firestore rules 或導入多老師 tenant 後，資料健康檢查必須限制在目前管理員所屬 `tenantId` 的資料內。
